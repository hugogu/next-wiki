# Add z.ai (Zhipu / GLM) AI Provider — Design

**Status:** Approved
**Date:** 2026-06-21
**Scope:** Register z.ai as a first-class AI provider vendor for chat, embedding, and image generation capabilities.

## Context

z.ai (Z.AI / Zhipu / BigModel) exposes an OpenAI-compatible HTTP API at
`https://api.z.ai/api/paas/v4` with **plain Bearer authentication** (no JWT, no
split-key signing):

- `POST /chat/completions` — streaming + non-streaming, multimodal input, tools,
  `reasoning_content` for thinking models (glm-5.x, glm-4.5+).
- `POST /images/generations` — returns `{ data: [{ url }] }` (temporary URL,
  30-day expiry). Models: `glm-image`, `cogview-4-250304`.
- `POST /embeddings` — standard OpenAI shape.
- `GET /models` — undocumented but expected to follow the OpenAI contract.

Reference docs:
- https://docs.z.ai/api-reference/llm/chat-completion
- https://docs.z.ai/api-reference/image/generate-image

## Decision

**Approach A — New vendor reusing the `openai_compatible` kind** (the `kimi`
precedent). No new adapter file, no new kind, no registry or http-client changes.

Rationale: z.ai's wire protocol is already fully handled by
`OpenAiCompatibleAdapter`:

| z.ai characteristic | Handled by |
|---|---|
| Bearer auth | `http-client.ts` default non-anthropic branch |
| Streaming chat SSE | `OpenAiCompatibleAdapter.streamText` |
| Image URL response | `OpenAiCompatibleAdapter.generateImage` URL branch (`openai-compatible.ts:202`) |
| Embeddings | `OpenAiCompatibleAdapter.embed` |
| `/models` discovery | `OpenAiCompatibleAdapter.listModels` / `testConnection` (fails gracefully → manual fallback) |

Rejected alternatives:
- **B. New `zai` adapter kind** (like `minimax`) — rejected; no protocol
  divergence exists to justify a subclass.
- **C. Reuse `custom` vendor only** — rejected; poor UX (no default base URL,
  no curated label, defeats the catalog).

## UX outcome

When an admin adds a new capability provider (chat / embedding / image), the
"平台" selector in `ProviderForm` will include a new **"Z.ai"** / **"Z.ai 智谱"**
option. Selecting it pre-fills the base URL with
`https://api.z.ai/api/paas/v4`. The admin pastes their API key, clicks Test,
then Create.

Type-safety guarantees the selector wiring stays in sync: `VENDOR_LABELS` is
`Record<AiProviderVendor, TranslationKey>`, so adding `'zai'` to the union
forces the label entry, the translation key, and both locale entries to be
added or the build fails.

## Out of scope (YAGNI)

- Surfacing `reasoning_content` from streaming — pre-existing limitation of
  `streamText`; not a regression.
- Multimodal (image/video/file) chat input — current `TextGenerationInput`
  messages are text-only by architecture.
- Downloading z.ai image URLs to permanent storage — the downstream pipeline
  already handles the `{ kind: 'url' }` output type.
- Pre-seeding GLM models — discovery handles it, manual fallback exists for
  image models and undiscovered chat models.
