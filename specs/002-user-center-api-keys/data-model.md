# Data Model: User Center & API Keys

**Feature**: `002-user-center-api-keys`
**Date**: 2026-06-18
**ORM**: Drizzle (PostgreSQL 16+). All schemas live in `apps/web/src/server/db/schema/`.
Migrations are idempotent and versioned under `apps/web/src/server/db/migrations/`.

This document describes **increments** to the 001 data model. Existing tables
(`spaces`, `users`, `sessions`, `pages`, `page_revisions`) are modified or
extended; new tables (`api_keys`, `api_audit_entries`) are added.

---

## New Enums

```ts
apiKeyScope = 'view' | 'create' | 'edit' | 'delete' | 'share' | 'run'
```

Added to `apps/web/src/server/db/schema/enums.ts`. Scopes are drawn from the
spec's predefined set (FR-007). `share` and `run` are forward-looking
reservations (A2) — they exist in the enum but have no enforcement targets in
this slice.

No new enum is needed for audit entries; the `status_code` is an integer and
the `auth_status` is a text field (`'authenticated' | 'invalid_key' |
'revoked_key' | 'disabled_user' | 'malformed_token'`).

## Modified Entity: `users`

The existing `users` table gains two new columns for server-side display
preferences (spec FR-005, clarification Q1).

| New Field | Type | Constraints | Notes |
|---|---|---|---|
| `theme_preference` | text | nullable | `'light' \| 'dark' \| 'auto'`; null = unset (use client default) |
| `locale_preference` | text | nullable | `'en' \| 'zh'`; null = unset (use cookie/Accept-Language default) |

Both columns are nullable so existing users are unaffected by the migration.
The User Center is the canonical place to set these (FR-005); client-side
`localStorage` serves only as a fast-init fallback.

**Migration**: `ALTER TABLE users ADD COLUMN theme_preference text, ADD COLUMN
locale_preference text;` — generated by `pnpm db:generate` (drizzle-kit).

No other columns on `users` change. `email`, `password_hash`, `display_name`,
`role`, `status`, `must_reset_password` remain as in 001.

## New Entity: `api_keys`

A personal credential for programmatic API access. Belongs to exactly one user.

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | uuid (pk) | default `gen_random_uuid()` | surrogate PK |
| `user_id` | uuid | fk → users(id), not null | owner |
| `name` | text | not null | user-supplied label (1–100 chars) |
| `scopes` | apiKeyScope[] | not null | immutable set chosen at creation (FR-008); stored as PostgreSQL `text[]` with app-level validation against the enum |
| `key_prefix` | text | not null | non-secret visible prefix (first 12 chars of key, e.g. `nwk_aB3xY9zK`) for identification + indexed lookup |
| `key_secret_encrypted` | text | not null | full key value encrypted with AES-256-GCM (includes nonce + tag, base64-encoded) — reversible on demand (FR-009, FR-010) |
| `created_at` | timestamptz | not null default `now()` | |
| `revoked_at` | timestamptz | nullable | null = active; set when user revokes (FR-011) |
| `last_used_at` | timestamptz | nullable | updated on successful auth (nice-to-have for key list display) |

**Indexes**:
- unique(`key_prefix`) — fast lookup during Bearer auth (prefix is unique by
  construction: 8 random base64url chars = 2^48 space).
- index(`user_id`, `revoked_at`) — list active keys for a user.
- index(`user_id`) — count keys per user for max enforcement (FR-015).

**Validation** (service layer, Zod-backed):
- `name`: 1–100 chars, non-empty after trim.
- `scopes`: non-empty array (at least one scope required, edge case); each
  element ∈ the `apiKeyScope` enum; duplicates rejected.
- Per-user active key maximum (FR-015): enforced in service, default 10.
- Key format: `nwk_` + base64url(32 random bytes) ≈ 48 chars. Generated
  server-side; user never provides the key value.

**Immutability** (FR-008): after insert, `scopes` and `name` are never updated.
The only mutable column is `revoked_at` (soft-delete equivalent for keys) and
`last_used_at` (operational metadata). To change scopes, generate a new key and
revoke the old one.

**Encryption details** (plan D3):
- Encryption: AES-256-GCM with a random 12-byte nonce per key.
- Key: `API_KEY_ENCRYPTION_KEY` env var (32-byte hex = 64 chars).
- Encrypted blob format: `base64(nonce || ciphertext || authTag)`.
- The `key_prefix` is NOT encrypted — it's a non-secret identifier.

## New Entity: `api_audit_entries`

An immutable record of a single API request made with (or attempting to use)
an API key. Append-only; no update or delete by normal operations (FR-020,
FR-021, spec A7).

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | uuid (pk) | default `gen_random_uuid()` | surrogate PK |
| `key_id` | uuid | fk → api_keys(id), nullable | null when key cannot be resolved (FR-021) |
| `user_id` | uuid | fk → users(id), nullable | null when user cannot be resolved (FR-021); equals key owner when resolved |
| `method` | text | not null | HTTP method (`GET`, `POST`, etc.) |
| `path` | text | not null | request path (e.g. `/api/pages/docs/intro`) |
| `status_code` | integer | not null | HTTP response status code (e.g. 200, 401, 403, 500) |
| `duration_ms` | integer | not null | request duration in milliseconds |
| `auth_status` | text | not null | `'authenticated' \| 'invalid_key' \| 'revoked_key' \| 'disabled_user' \| 'malformed_token'` |
| `error_message` | text | nullable | short error message on 4xx/5xx; null on success |
| `created_at` | timestamptz | not null default `now()` | when the request was processed |

**Indexes**:
- index(`user_id`, `created_at desc`) — user's own audit log (FR-022).
- index(`created_at desc`) — admin global audit log (FR-023).
- index(`key_id`, `created_at desc`) — filter by key.
- index(`status_code`) — filter by success/error (admin filter by status).

**Immutability**: rows are insert-only. No UPDATE or DELETE is issued by normal
operations. A future retention policy may add a scheduled cleanup (spec A7 —
not in this slice).

**Auth status mapping**:

| Scenario | `auth_status` | `key_id` | `user_id` | `status_code` |
|---|---|---|---|---|
| Valid key, successful request | `authenticated` | resolved | resolved | 2xx/4xx/5xx |
| Bearer token doesn't match any key | `invalid_key` | null | null | 401 |
| Bearer token matches a revoked key | `revoked_key` | resolved | resolved | 401 |
| Bearer token matches a key owned by a disabled user | `disabled_user` | resolved | resolved | 401 |
| Malformed/truncated Bearer token | `malformed_token` | null | null | 401 |

For `authenticated` entries with a 4xx/5xx response, `error_message` captures
a short description (e.g. `"FORBIDDEN"`, `"NOT_FOUND"`, `"VALIDATION_ERROR"`).
For 2xx responses, `error_message` is null.

## Validation Rules (service-layer, Zod-backed)

- **Create API key**: `name` (1–100 chars); `scopes` (non-empty array, each ∈
  `apiKeyScope` enum, no duplicates); per-user active key count < max (10).
- **Reveal API key**: user must own the key; key must exist; returns decrypted
  secret. Rate-limited client-side (explicit click only, FR-009).
- **Revoke API key**: user must own the key; sets `revoked_at = now()`.
- **Update profile (nickname)**: `displayName` (1–100 chars, or null to clear).
- **Change email**: `email` (valid format, unique — rejects duplicates with
  CONFLICT, FR-003). No current-password re-auth (clarification Q3).
- **Change password**: `currentPassword` (must match); `newPassword` (min 8
  chars). Incorrect current password rejected (FR-004).
- **Update preferences**: `theme` ∈ `['light', 'dark', 'auto']` or null;
  `locale` ∈ `['en', 'zh']` or null.

## State Transitions

### API Key lifecycle

```text
[created] ──active──> [revoked]
     │                    │
     └─ created_at set    └─ revoked_at set; subsequent requests return 401
```

- `active → revoked`: by owner (revoke action) or admin (future). No
  reactivation; a revoked key is permanently invalid.
- `last_used_at` updates on each successful auth but does not change lifecycle
  state.
- Keys do not expire (FR-014, A5). No TTL field exists in this slice.

### User preferences

```text
(null) ──set──> 'light' | 'dark' | 'auto'  (theme)
(null) ──set──> 'en' | 'zh'                 (locale)
         └──clear──> (null)
```

Preferences are freely set and cleared. Setting a preference to null reverts
to the client-side default (localStorage / system preference).

## Permission Resolution (increment)

The `can(ctx, action, resource)` chokepoint is extended to handle the `api_key`
actor kind (plan D2). The existing permission matrix from 001 is unchanged for
`user` and `anonymous` actors. For `api_key` actors:

1. **Scope check**: map the requested `action` to a scope (D2 mapping table). If
   the key's scopes do not include the mapped scope → **deny** (scope
   insufficient).
2. **Role check**: if the scope check passes, evaluate the same role + authorship
   rules as a `user` actor. If the role denies → **deny** (role insufficient).
3. **`manage_users`**: never allowed via API key (no scope maps to it).

This implements scope ∩ role: both checks must pass for the action to be
allowed.

| Action | Required Scope | Notes |
|---|---|---|
| `read` | `view` | List/read published pages |
| `read_draft` | `view` | Read draft revisions (still constrained by authorship) |
| `create` | `create` | Create new pages |
| `edit` | `edit` | Create new draft revisions |
| `publish` | `edit` | Publish a draft (treated as an edit operation) |
| `delete` | `delete` | Soft-delete pages |
| `manage_users` | *(none)* | Never allowed via API key |

## Forward-compatibility (no migration needed to add later)

- `api_keys.expires_at` (nullable timestamptz) → key TTL/expiration (A5). Add
  column + check in auth resolver. No migration impact on existing rows (null =
  no expiration).
- `api_audit_entries` retention policy → a scheduled pg-boss job that deletes
  entries older than N days. The table is already append-only; a DELETE query
  is the only addition needed.
- New scopes → add to the `apiKeyScope` enum + update `can()` mapping. Existing
  keys are unaffected (their stored scope set is a subset of the expanded enum).
- Key rotation → a re-encryption script that reads all `key_secret_encrypted`,
  decrypts with the old key, re-encrypts with the new key. No schema change.
