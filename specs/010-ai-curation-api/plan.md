## Summary

The AI Curation API extends the v1 public REST surface (and the `@next-wiki/mcp-server` companion) with five capabilities that close the loop between AI agents and the wiki as a knowledge store:

1. **Keyword search (extended)** — frontmatter filters added to `GET /api/v1/search/pages`; permission filtering and response shape are regression-safe.
2. **Semantic search (new)** — `POST /api/v1/search/semantic` + `GET /api/v1/search/semantic/{id}`, wrapped around the existing `ai_actions` lifecycle with a new `ai.read` API-key scope.
3. **Structured frontmatter** — `frontmatter: object|null` field on every page/list/search/tree/revision response; frontmatter-based filters on list and search.
4. **Link graph traversal** — `GET /api/v1/pages/{id}/links` (outbound, classified) and `GET /api/v1/graph/neighbors` (bounded multi-hop).
5. **Bulk write operations** — `POST /api/v1/pages/batch/update` and `POST /api/v1/pages/batch/delete`, both with `dry_run=true` and per-item partial-success reporting.

A single **permission-hardening change** is bundled: the pre-existing `retrieve()` function in `apps/web/src/server/services/ai-retrieval.ts:34` gains a `PermCtx` parameter and applies the `can(ctx, 'read', { kind: 'page_list' }, { anonymousRead })` filter, closing a real data-leak vector that affects both the new public semantic search and the existing in-app Q&A vector path.

Technical approach: thin v1 route handlers in `apps/web/app/api/v1/` (existing `withPublicApi` + `withApiAudit` + `parsePublicJson` / `parsePublicQuery` patterns from `_shared/route.ts`), backed by an extension of the `public-content` facade (plus a small new `public-ai` facade for the AI-action wrapping), with no schema migrations, no new default services, no new tables, and no new runtime dependencies. OpenAPI document is regenerated by `pnpm --filter @next-wiki/web openapi:generate`. MCP tool surface mirrors the v1 surface 1:1.

## Technical Context

**Language/Version**: TypeScript 5.6 on Node.js 20.9+ (Docker image tracks Node 24; Next.js 16 runtime floor)
**Primary Dependencies**: Next.js 16, React 19.2, Drizzle ORM, pg-boss, Zod, `unified`/`remark`/`unist-util-visit`, `@next-wiki/mcp-server` (existing). No new top-level runtime dependencies required — the `yaml` package (`apps/web/package.json:59`) already covers frontmatter parsing.
**Storage**: PostgreSQL 16 with pgvector 0.8.x. No new tables; the spec explicitly forbids them (FR-029). Frontmatter is parsed from `page_revisions.content_source` at response time; outbound links are extracted from the same content at response time. These are derived projections, not stored columns.
**Testing**: Vitest (unit + integration, existing `apps/web/src/server/**/*.test.ts` pattern) + Playwright (e2e, existing `apps/web/e2e/` pattern). Three specific test patterns are mandated: (a) `permission boundary` test asserting the new `ai.read` scope, mirroring `apps/web/src/server/permissions/ai-permissions.test.ts:14`; (b) `no-leakage` test asserting that an unreadable page never appears in any list response, mirroring `apps/web/src/server/services/public-content-read.test.ts:65-81`; (c) `regression-suite` test that the Q&A vector path also no longer leaks, mirroring the new permission filter.
**Target Platform**: Linux server (default deployment), Docker Compose. No platform-specific concerns.
**Project Type**: Web service — Next.js 16 App Router, RSC + REST route handlers under `apps/web/app/api/`. The feature is purely server-side: no frontend UI changes, no client component changes. The one UI touch is adding the new `ai.read` scope to the API-key creation dialog (`apps/web/src/components/user-center/ApiKeyCreateDialog.tsx:11`).
**Performance Goals** (from spec SC-002, SC-003): 2-hop neighborhood traversal ≤ 1 second on a 10,000-page wiki; batch update + verify read ≤ 5 seconds for 50 items. Keyword search regression: byte-compatible with current implementation, no new DB scans. Frontmatter parsing adds ~µs per page (`yaml` package, no schema validation, no JS-type risk per research §5).
**Constraints** (Constitution P1, P5, P7, P8, P9): No new default runtime service (P1); every endpoint and every result list enforces permission checks at the `can(ctx, action, resource, opts)` chokepoint (P5); no synchronous LLM calls in HTTP request handlers — semantic search runs through the existing `ai_actions` + pg-boss pipeline (P7); every mutation creates an immutable revision (P8); all public-API additions flow through next-open-api (P9).
**Scale/Scope**: 10,000 pages max (spec SC-003), 50 items per batch (existing `batchCreatePages` cap, FR-019/FR-020), no new tables. The `ai_actions` table already accommodates the new `feature='semantic_search'` use; no schema migration. One pgEnum change (add `ai.read` to `apiKeyScopeEnum`).

No `NEEDS CLARIFICATION` items remain — every unknown has been resolved by [`research.md`](./research.md) with file:line citations.

## Constitution Check

*Gate: must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Evidence |
|---|---|---|
| **P1** Simple Deployment, Personal by Default | **PASS** | FR-030 prohibits new default runtime dependency; the spec only adds code, no new services. The `ai.read` scope is an enum value + a UI checkbox, not a service. The new `public-ai` facade is server-only code that compiles into the same Next.js image. |
| **P2** AI-Native Creation, Never Vendor-Locked | **PASS** | Semantic search reuses the existing provider-agnostic `createAiProviderAdapter` registry (`apps/web/src/server/ai/registry.ts:17`) and the existing `ai_settings` provider configuration. No vendor SDK is hard-coded. |
| **P3** KB is Portable AI Memory | **PASS** | FR-005 mandates grounded citations; FR-007 forbids silent fallback; FR-009 closes the pre-existing pgvector-without-permission leak documented in [`research.md` §5](./research.md). The retrieval function is refactored to take a `PermCtx` and apply the `can(ctx, 'read', { kind: 'page_list' }, { anonymousRead })` filter (the same pattern as `full-context.ts:55-58`), bringing the vector path to parity with the full-context path. |
| **P4** Rendering Pipeline is Sacred | **N/A** | The spec does not touch the rendering pipeline. No new renderers, transformers, or plugins. |
| **P5** Permissions First-Class | **PASS** | FR-009 is a dedicated cross-endpoint permission-filter requirement. The new `ai.read` scope maps to `use_ai_search` and (future) `use_ai_qa`; `use_ai_search` and `use_ai_qa` are removed from the api-key hard-deny list at `apps/web/src/server/permissions/index.ts:132-144` so the new scope can take effect. FR-009 closes the existing leak; FR-011/FR-014/FR-019/FR-025 all enforce per-result filtering. |
| **P6** Style System Independence | **N/A** | No UI work in scope. The one UI touch (api-key scope checkbox) uses the existing `ApiKeyCreateDialog` design tokens. |
| **P7** Async-First for Heavy Operations | **PASS** | FR-004 mandates async for semantic search; FR-018 (graph neighborhood) is bounded by `depth ∈ [1,3]` so a 10k-page wiki fits in <1s; FR-020 (batch update) is per-item synchronous because each item is a single revision (well under 500ms), but the route returns a single 200 with per-item results, not a job. The `dry_run=true` mode (FR-022) is sync and zero-write. |
| **P8** Version Everything | **PASS** | FR-024 mandates a new revision per title/path/frontmatter change; the existing `pages.ts::newDraft` revision-creation pattern (L345-430) is reused. Batch soft-delete (FR-021) reuses `pages.ts::remove` (L235-265), which is the canonical soft-delete. No hard delete. |
| **P9** Open Standards | **PASS** | FR-026 mandates next-open-api regeneration; FR-027 mandates MCP tool exposure for every new endpoint, with the existing naming convention (`search_*`, `list_*`, `get_*`, `batch_*`). The 17 existing v1 routes are the precedent; the 5 new routes follow exactly. |
| **P10** Explicit Over Implicit | **PASS** | The keyword/semantic split is itself a P10 application (two distinct resources rather than a `mode` flag, FR-010). The link `source` discriminator is explicit. `dry_run` is an explicit flag (FR-022). The `ai.read` scope is explicit, with a documented `use_ai_search` / `use_ai_qa` mapping in spec Assumption 10. |
| **P11** Native Web Navigation | **N/A** | No URL changes. The existing `/api/v1/...` path space is reused; no verb-style URLs. |

**Anti-pattern check** (research §"Cross-cutting observations"):

- **AI content as second-class**: PASS — frontmatter and links are derived at query time from the same `content_source` column; no second table. AI-indexed pages and human-authored pages are indistinguishable to the storage and permission layers (already the case).
- **Vendor-locked AI integration**: PASS — semantic search reuses the existing `createAiProviderAdapter` factory and the same `ai_providers` table; no new SDK.
- **Editor format stored as rendered HTML**: N/A — no editor work.
- **State without a URL**: N/A — server-side only.
- **Duplicate feature entry points**: PASS — `submit_semantic_search` and `get_semantic_search_results` are the only entry points; the existing internal `/api/ai/searches` route is session-only and is not duplicated.
- **Per-page bespoke styling**: N/A — no UI.

**Re-evaluation after Phase 1 design**: no principle violations. All gates pass.

## Implementation Order (dependency-driven)

The five capability groups are not all independent. The dependency graph (Phase 2 tasks will elaborate):

```
1. ai.read scope + permission map        ← no dependency, unblocks everything
2. shared retrieve() permission filter  ← depends on (1) (only the api-key branch needs the filter; user/admin was already correct via full-context)
3. frontmatter parser extraction        ← unblocks (4), (5)
4. frontmatter in page responses + filters  ← depends on (3)
5. outbound links + graph               ← depends on (3) (frontmatter-aware link extraction) and (1) (read filter)
6. batch update + delete + dry_run      ← depends on (1) and reuses existing revision/soft-delete paths
7. semantic search submit + poll        ← depends on (1), (2) (the permission filter), and the new scope
8. MCP tool surface (mirrors 4–7)       ← depends on (4–7) being available over v1
9. OpenAPI regeneration + finalization  ← depends on (4–7) JSDoc + (shared schemas)
```

Within each capability, the order is: shared Zod schema → facade function → route handler → route test → facade test → MCP tool (if applicable) → OpenAPI regeneration. This is the same order used by the existing 008-public-wiki-api-maintenance work.
