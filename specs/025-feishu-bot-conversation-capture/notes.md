# Implementation Notes: Feishu Bot Conversation Capture (025)

**Status**: Implementation complete (Phases 1–9). Ready for `docker compose up -d --build` full-stack verification and quickstart.md walkthrough.

## Summary for the next agent

- **AI Conversations toggle**: `AI_CONVERSATIONS_SOURCE_KEY = 'ai-conversations'` (`packages/shared/src/content-data-sources.ts`) is now the only Admin-facing Data Source. It lives under `/admin/bots?tab=general` (`BotsTabs.tsx` → `ContentDataSourcesPanel`). `/admin/content` is a redirect to that URL, not a second editor. The legacy `WIKI_AI_CONVERSATIONS_SOURCE_KEY = 'wiki-ai-conversations'` is a read-only alias: `content-data-sources.ts::isDataSourceEnabled` / `listDataSources` lazily copy its `enabled` state into the new key on first read, then never touch it again.
- **Feishu captures reuse the 023 pipeline unchanged**: `feishu-delegation.ts` already called `createWikiQuestion(..., requestMetadata: { origin: 'feishu', feishuSessionId })`; no change was needed there. `raw-conversations.ts::captureConversation` now infers `channel` (`'feishu'` | `'wiki-ai'`) from `action.requestMetadata.origin` via `resolveConversationChannel` and stamps it into `source_metadata.channel` on every captured revision.
- **`RawConversationPointer.channel`**: resolved in `ai-actions.ts::resolveRawConversationPointer(s)` by reading the captured page's current published revision's `source_metadata.channel` (defaults to `'wiki-ai'` when absent/invalid). Flows automatically into `/api/ai/sessions` and `/api/ai/sessions/[id]`.
- **Search**: `candidate-projection.ts` now also selects `page_revisions.source_metadata` and derives `PublicPageResource.conversationChannel` for Conversation-category candidates only. No branch was needed in `coordinator.ts` — permission projection and fusion are channel-agnostic by construction (D6). `HeaderHybridSearch.tsx` renders a "Feishu" chip (`header.search.source.feishu` i18n key, shared with the Chat History badge) when `conversationChannel === 'feishu'`.
- **UI badge**: `ConversationSessionView` takes an optional `channel` prop and renders a small "Feishu" badge next to the status badge; omitted for `'wiki-ai'`/absent. Wired from `AiSessionsPanel.tsx` via `viewing.rawConversation?.channel` (no plumbing through `ReconstructedSession` needed — the list/detail pointer already carries it).
- **Bot Session stays a thin wrapper**: no schema change. `feishu-sessions.ts` now has a header comment formalizing the contract (D3); `feishu-sessions.wrapper.test.ts` enforces it (column check + upsert-not-duplicate behavior).
- **Audit**: `jobs/raw-conversation-capture.ts::runRawConversationCapture` now writes an `api_audit_entries` row after every successful capture, with `origin` mirrored from the capture's `channel` (`'feishu'` → `'feishu'`, else `'web'`). This is a new call site — the 023 code had no capture-level audit write despite the plan assuming one existed (D4); `CaptureOutcome`'s `captured` variant was extended with `channel`/`actorUserId`/`correlationId` to carry what the job wrapper needs without a second DB query.
- **MCP server**: `packages/mcp-server` exposes `conversationChannel` on `search_wiki` results (its own local `publicPageResourceSchema` mirrors the shared one, so it needed a matching field addition) and the tool description now mentions it.

## Deviations from plan.md worth knowing about

1. **i18n key paths differ from tasks.md's guess.** The real Admin panel copy lives at `admin.contentDataSources.sources.aiConversations.{label,description}` (not `dataSources.content.aiConversations`), and the Feishu badge/chip key is `header.search.source.feishu` (shared with the existing `header.search.source.conversation` key), not a separate `chat.history.feishuBadge.label` — the UI contract explicitly says the chip and badge should share one key, and `header.search.source.*` was the only pre-existing home for that kind of source chip.
2. **`/admin/content` nav entry removed, not redirected-with-link.** Bots already has its own nav entry; keeping a second "Data Sources" nav item pointing at the same destination would itself be a duplicate entry point (P11), so `Navigator.tsx`'s Content group lost the item entirely. The route itself still redirects (`next/navigation` `redirect()`) so old bookmarks/links don't 404.
3. **Audit write lives in `jobs/raw-conversation-capture.ts`, sourced from `captureConversation`'s outcome, not a separate DB read.** `CaptureOutcome`'s `captured` case now carries `channel`, `actorUserId`, and `correlationId` so the job wrapper needs no extra query. `ai-cleanup.ts`'s pre-expiry capture path was deliberately left unchanged (does not write the audit entry) — out of the literal task scope; flag if that gap matters product-wise.
