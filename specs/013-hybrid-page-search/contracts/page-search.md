# Existing Page-Search Resource Contract

**Phase 1 output** | **Date**: 2026-07-10

This feature extends the existing `/api/v1/search/pages` resource only. It introduces no new search path.

## Compatibility rule

`GET /api/v1/search/pages` remains the current keyword-search operation.

- Existing query parameters, response shape, permissions, pagination, and keyword ranking remain compatible.
- It remains a non-mutating, idempotent GET.
- Existing clients are not required to send any new field and do not receive an unexpected top-level envelope change.

## Hybrid query operation

### `POST /api/v1/search/pages`

Starts or resumes one Header hybrid-search attempt. It is idempotent for the same `searchRecordId` and authenticated/anonymous session owner.

### Request: `kind: "query"`

```json
{
  "kind": "query",
  "searchRecordId": "uuid",
  "searchSessionId": "uuid",
  "q": "authentication architecture",
  "limit": 20
}
```

| Field | Rules |
|---|---|
| `kind` | Required literal `query`. |
| `searchRecordId` | Required UUID generated by the Header for this qualified input attempt. Repeating it is a retry, not a new search. |
| `searchSessionId` | Required UUID generated when the overlay opens. It binds anonymous requests to the same short-lived interaction. |
| `q` | Required trimmed query, 2–200 characters. |
| `limit` | Optional integer 1–20, defaults to 20. |

### Response: 200

```json
{
  "searchRecordId": "uuid",
  "semanticState": "pending",
  "items": [
    {
      "page": {
        "id": "uuid",
        "path": "docs/auth",
        "title": "Authentication",
        "status": "published"
      },
      "excerpt": "…token rotation and session lifetime…",
      "score": 0.031,
      "matchSources": ["keyword"]
    }
  ]
}
```

| Field | Rules |
|---|---|
| `searchRecordId` | Echoes the accepted request ID. |
| `semanticState` | `pending`, `ready`, `unavailable`, `failed`, or `skipped`. It is a non-sensitive coverage state, not an index/provider diagnostic. |
| `items` | One ranked, de-duplicated list of readable pages. Empty is valid. |
| `items[].page` | Existing compact public page shape; raw content is never returned. |
| `items[].excerpt` | A relevant keyword or semantic excerpt; no full page body. |
| `items[].score` | Fused ordering score for this hybrid operation; clients must not compare it with legacy GET scores. |
| `items[].matchSources` | One or both of `keyword`, `semantic`; informs UI explanation without exposing unavailable candidates. |

When `semanticState` is `pending`, the Header repeats the exact same idempotent query request with bounded backoff. When it becomes `ready`, the result contains the merged vector contribution. When it is `unavailable`, `failed`, or `skipped`, the returned keyword results remain usable and the UI shows a generic reduced-coverage message.

### Authorization and non-disclosure

- Keyword candidates require the existing page-read permission.
- Semantic enrichment is attempted only for an actor that may use semantic search and only after page-read authorization is established. API-key callers need the current relevant search scopes; session callers use their existing entitlement/permission context.
- Anonymous visitors and actors without semantic permission receive keyword-only results with a non-sensitive state. The response must not disclose whether an index, provider, or protected candidate exists.
- A page returned by either source must pass the same visibility check before title, excerpt, score, or count is produced.

### Idempotency and errors

- Repeating a query request with the same `searchRecordId`, same owner/session, and same normalized payload returns the same record/lifecycle rather than creating another row or action.
- Reusing the ID with another actor/session or different normalized query is rejected as a validation/conflict error with no cross-user data disclosure.
- Invalid query/body returns the standard public validation response.
- Analytics persistence failure is logged and does not change an otherwise valid search response.

## Behavior operation

### `POST /api/v1/search/pages`

Records one explicit result-open or Escape outcome on the same existing resource path.

### Request: `kind: "behavior"`

```json
{
  "kind": "behavior",
  "eventId": "uuid",
  "searchRecordId": "uuid",
  "searchSessionId": "uuid",
  "action": "result_open",
  "pageId": "uuid"
}
```

| Field | Rules |
|---|---|
| `kind` | Required literal `behavior`. |
| `eventId` | Required client-generated UUID; primary idempotency key. |
| `searchRecordId`, `searchSessionId` | Required and must belong to the requester/session. |
| `action` | `result_open` or `escape`. |
| `pageId` | Required only for `result_open`; absent for `escape`. The server confirms it remains readable. |

### Response

`204 No Content` on first acceptance and on an idempotent duplicate. The Header sends this best-effort with `keepalive` and must not wait before closing or navigating.

### Behavior safeguards

- A behavior cannot be attached to another user or anonymous session's search record.
- `result_open` only records a currently readable page; it never creates a successful behavior for a protected or missing page.
- Repeated keydown, click, render, or transport retry with the same `eventId` produces no duplicate row.
- The client emits only one terminal behavior per overlay session; the server still enforces event idempotency.

## OpenAPI and shared-schema work

The implementation adds shared Zod request/response schemas in `packages/shared/src/pages.ts`, described counterparts in `apps/web/src/server/api/openapi-schemas.ts`, route annotations, and regenerated OpenAPI output. The route handler remains a thin adapter over service-layer functions.
