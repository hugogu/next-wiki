# Quickstart: Web Analytics Integrations

**Feature**: 024-analytics-integrations
**Date**: 2026-07-21

This is a runnable validation guide. It documents the scenarios that prove the
feature works end-to-end. Implementation details belong in `tasks.md` (Phase
2). Keep this document as a validation/run guide.

For contract details, see [contracts/api.md](./contracts/api.md),
[contracts/script-injection.md](./contracts/script-injection.md), and
[contracts/admin-ui.md](./contracts/admin-ui.md). For data model details,
see [data-model.md](./data-model.md).

## Prerequisites

1. **PostgreSQL 16** running (Docker Compose is the canonical local setup):

   ```bash
   docker compose up -d --build
   ```

   This brings up the DB and applies migrations (including the new
   `0030_*.sql` generated by `pnpm db:generate`).

2. **Migration applied**: after the schema changes in
   `src/server/db/schema/enums.ts` and `src/server/db/schema/index.ts`, run:

   ```bash
   pnpm db:generate
   pnpm db:migrate
   ```

   Confirm the second `pnpm db:generate` reports "No schema changes, nothing
   to migrate".

3. **Dev server** running:

   ```bash
   pnpm dev
   ```

4. **Admin user**: sign in as an admin (the first-run onboarding creates one).
   If no admin exists, run the onboarding flow at `/setup`.

5. **OpenAPI regenerated** (after the new `/api/settings/analytics` route is
   added):

   ```bash
   pnpm --filter @next-wiki/web openapi:generate
   ```

## Scenario 1: Admin configures Baidu Tongji

**Goal**: Verify an admin can enable Baidu Tongji and the script appears on all
pages.

1. Sign in as admin.
2. Navigate to `/admin/analytics` (or click the new "Analytics" entry in the
   admin nav under the "System" group).
3. In the Baidu Tongji card:
   - Toggle "Enable" on.
   - Enter a valid Tracking ID (e.g. `abcdef0123456789abcdef0123456789` - a
     32-character hex string; Baidu's real ID format).
4. Click "Save".
5. **Expected**: a success message "Analytics settings saved." appears inline.

**Validation**:
- The DB row `analytics_provider_settings` for `provider = 'baidu_tongji'`
  has `enabled = true` and `tracking_id = 'abcdef0123456789abcdef0123456789'`.
- `curl -b "session=<admin-session-cookie>" http://localhost:3000/api/settings/analytics`
  returns a JSON object where `providers[0].enabled = true` and
  `activeScriptContent` contains
  `hm.baidu.com/hm.js?abcdef0123456789abcdef0123456789`.

## Scenario 2: Script injected on every page surface

**Goal**: Verify the framework-level injection point covers all surfaces
(reader, editor, admin, auth, public home).

Prerequisite: Scenario 1 completed (Baidu Tongji enabled).

1. Open each of the following URLs in a browser and view the page source
   (or use `curl` and `grep`):

   ```bash
   # Public home
   curl -s http://localhost:3000/ | grep -c "hm.baidu.com/hm.js"
   # Reader page (use any existing page slug)
   curl -s http://localhost:3000/s/<your-slug> | grep -c "hm.baidu.com/hm.js"
   # Editor (requires session - use a logged-in browser)
   # Admin page
   curl -s -b "session=<your-session-cookie>" http://localhost:3000/admin/analytics | grep -c "hm.baidu.com/hm.js"
   # Auth page (login)
   curl -s http://localhost:3000/auth/login | grep -c "hm.baidu.com/hm.js"
   ```

2. **Expected**: every surface returns `1` (the provider loader URL appears
   exactly once on each page). The loader is configured with the saved Tracking
   ID.

**Validation**:
- The `<script id="app-analytics">` tag appears in the `<head>` of every
  rendered page, immediately after the existing `themeScript` inline
  `<script>`, and its content contains the enabled provider loader URL.
- A code audit (`rg "hm.baidu.com|googletagmanager" apps/web/app/ apps/web/src/`)
  finds the vendor host references in **only** the service file
  `apps/web/src/server/services/analytics.ts` - never in any page or layout
  component.

## Scenario 3: Disable a provider

**Goal**: Verify disabling a provider removes the script from all pages and
retains the Tracking ID.

Prerequisite: Scenario 1 completed (Baidu Tongji enabled with Tracking ID).

1. Navigate to `/admin/analytics`.
2. In the Baidu Tongji card, toggle "Enable" off.
3. Click "Save".
4. **Expected**: success message appears. The Tracking ID field still shows
   the previously entered value (it is retained for re-enablement).

**Validation**:
- The DB row for `baidu_tongji` has `enabled = false` and
  `tracking_id = 'abcdef0123456789abcdef0123456789'` (retained).
- `curl http://localhost:3000/ | grep -c "hm.baidu.com/hm.js"` returns `0`
  (the script is absent on all pages).
- `curl -b "session=<admin-session-cookie>" http://localhost:3000/api/settings/analytics`
  returns `providers[0].enabled = false` and `activeScriptContent = ""`.

---

## Scenario 4: Re-enable retains Tracking ID

**Goal**: Verify the retained Tracking ID works when re-enabling.

Prerequisite: Scenario 3 completed (Baidu Tongji disabled, Tracking ID
retained).

1. Navigate to `/admin/analytics`.
2. In the Baidu Tongji card, toggle "Enable" on (do NOT clear the Tracking
   ID field).
3. Click "Save".
4. **Expected**: success message appears. The script reappears on all pages
   with the retained Tracking ID.

**Validation**:
- `curl http://localhost:3000/ | grep -c "hm.baidu.com/hm.js"` returns `1`.
- The Tracking ID in the rendered script matches the saved value.

## Scenario 5: Validation rejects invalid Tracking ID

**Goal**: Verify the service rejects enabling a provider with an invalid
Tracking ID.

1. Navigate to `/admin/analytics`.
2. In the Google Analytics card:
   - Toggle "Enable" on.
   - Enter an invalid Tracking ID (e.g. `not-a-valid-id`).
3. Click "Save".
4. **Expected**: an error message "Tracking ID does not match the expected
   format." appears inline. The form state is unchanged (Google Analytics
   remains disabled).

**Validation**:
- `curl -b "session=<admin-session-cookie>" http://localhost:3000/api/settings/analytics`
  returns `providers[1].enabled = false` (Google Analytics is still disabled).
- The DB row for `google_analytics` either does not exist or has
  `enabled = false` (the previous state is preserved).

## Scenario 6: Both providers enabled simultaneously

**Goal**: Verify multiple providers' scripts are injected together.

1. Sign in as admin, navigate to `/admin/analytics`.
2. Enable Baidu Tongji with Tracking ID `abcdef0123456789abcdef0123456789`.
3. Enable Google Analytics with Tracking ID `G-A1B2C3D4E5`.
4. Click "Save".
5. **Expected**: success message appears. Both scripts appear on every page.

**Validation**:
- `curl http://localhost:3000/ | grep -c "hm.baidu.com/hm.js"` returns `1`.
- `curl http://localhost:3000/ | grep -c "googletagmanager.com/gtag/js"` returns `1`.
- Both scripts are present in the `<head>` of every page, in a stable order
  (Baidu first, then Google - the registry order).

## Scenario 7: Non-admin user is denied

**Goal**: Verify the admin page and the PUT endpoint are admin-only.

1. Sign in as a non-admin user (editor or reader).
2. Navigate to `/admin/analytics`.
3. **Expected**: 404 (the existing pattern for admin-only pages - the page
   calls `notFound()` when the permission check fails).
4. Attempt `GET /api/settings/analytics` with the non-admin session:

   ```bash
   curl -b "session=<editor-session>" http://localhost:3000/api/settings/analytics
   ```

5. **Expected**: `403 FORBIDDEN`.

6. Attempt `PUT /api/settings/analytics` with the non-admin session:
   ```bash
   curl -X PUT -b "session=<editor-session>" \
     -H "Content-Type: application/json" \
     -d '{"providers":[{"provider":"baidu_tongji","enabled":true,"trackingId":"abc"}]}' \
     http://localhost:3000/api/settings/analytics
   ```
7. **Expected**: `403 FORBIDDEN` with a message "You do not have permission
   to manage site settings".

## Scenario 8: Anonymous visitor cannot read or mutate settings

**Goal**: Verify the settings API requires an admin session. Anonymous visitors
receive analytics only through page HTML, not through the configuration API.

1. Attempt `GET /api/settings/analytics` without a session:

   ```bash
   curl http://localhost:3000/api/settings/analytics
   ```

2. **Expected**: `401 UNAUTHORIZED`.

3. Attempt `PUT /api/settings/analytics` without a session:
   ```bash
   curl -X PUT \
     -H "Content-Type: application/json" \
     -d '{"providers":[{"provider":"baidu_tongji","enabled":true,"trackingId":"abc"}]}' \
     http://localhost:3000/api/settings/analytics
   ```

4. **Expected**: `401 UNAUTHORIZED`.

**Note**: The root layout reads the active script content through the
server-side `getActiveAnalyticsScriptContent()` service, not through the HTTP
settings API.

## Scenario 9: Public page cache is revalidated on mutation

**Goal**: Verify that changing the analytics configuration invalidates the
site-shell cache so the next request sees the updated script set.

1. With Baidu Tongji enabled, `curl http://localhost:3000/` twice. The
   second response is served from the ISR cache (same `x-nextjs-cache` header
   value, faster response time). Confirm `hm.baidu.com/hm.js` is present.
2. Disable Baidu Tongji via the admin UI (or `PUT /api/settings/analytics`).
3. `curl http://localhost:3000/` again.
4. **Expected**: the response no longer contains `hm.baidu.com/hm.js`. The
   cache was invalidated by `invalidateSiteShellCache()` called during the
   mutation.

**Validation** (in test mode, where `shouldUseDataCache()` returns false):
- The unit test `analytics.test.ts` asserts that `updateAnalyticsProvider`
  calls `invalidateSiteShellCache` after the DB write. (This is a behavioral
  assertion via a mock or spy - see the existing `site-settings.test.ts` for
  the pattern.)

## Scenario 10: New page automatically gets the script

**Goal**: Verify the framework-level injection covers new routes without
per-page code.

Prerequisite: Baidu Tongji enabled.

1. Add a new page or route to the app (e.g. a new admin page, or a new
   public route) **without** any analytics-specific code in it.
2. Open the new page in a browser.
3. **Expected**: the Baidu Tongji script is present in the page source,
   without the new page having to reference analytics.

This is a design-time validation: the root `app/layout.tsx` wraps every
route, so any new route automatically inherits the analytics script. The
scenario is included to make the contract explicit and to catch regressions
if someone accidentally removes the injection from the root layout.

## Running the automated tests

After implementation, run:

```bash
# Unit + integration tests
pnpm --filter @next-wiki/web test -- analytics

# E2E tests (Playwright) - requires the dev server running
pnpm --filter @next-wiki/web test:e2e -- analytics

# Lint + typecheck
pnpm --filter @next-wiki/web lint
pnpm --filter @next-wiki/web typecheck

# OpenAPI regeneration (no-op if no route changes)
pnpm --filter @next-wiki/web openapi:generate
```

All tests must pass before the feature is considered complete.

## Cleanup

To reset the analytics configuration after testing:

```sql
-- Connect to the PostgreSQL DB (e.g. via psql or the DB admin tool)
TRUNCATE TABLE analytics_provider_settings;
```

Or via the admin UI: disable every provider and save. The Tracking IDs will
be retained (per Scenario 3); a full reset requires the SQL above.
