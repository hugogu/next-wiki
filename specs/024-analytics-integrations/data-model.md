# Data Model: Web Analytics Integrations

**Feature**: 024-analytics-integrations
**Date**: 2026-07-21

This document describes the new database entities introduced by the feature.
All schema changes are additive and generated by `pnpm db:generate` (never
hand-authored - see `CLAUDE.md` "Database Migrations (Drizzle)").

---

## New Enum: `analytics_provider`

**File**: `apps/web/src/server/db/schema/enums.ts`

```ts
export const analyticsProviderEnum = pgEnum('analytics_provider', [
  'baidu_tongji',
  'google_analytics',
]);
```

**Postgres type name**: `analytics_provider` (snake_case, per project convention).

**Values**:
- `baidu_tongji` - Baidu Tongji
- `google_analytics` - Google Analytics (GA4)

**Mirrored Zod enum** in `packages/shared/src/analytics.ts`:

```ts
export const analyticsProviderSchema = z.enum([
  'baidu_tongji',
  'google_analytics',
]);
export type AnalyticsProvider = z.infer<typeof analyticsProviderSchema>;
```

The shared enum and the DB enum MUST stay in sync. Adding a new provider
requires appending to both arrays in the same commit.

## New Table: `analytics_provider_settings`

**File**: `apps/web/src/server/db/schema/index.ts`

Key-addressed registry of per-provider analytics configuration. One row per
registered provider. Mirrors the `content_data_source_settings` pattern.

### Schema (Drizzle)

```ts
export const analyticsProviderSettings = pgTable(
  'analytics_provider_settings',
  {
    provider: analyticsProviderEnum('provider').primaryKey(),
    enabled: boolean('enabled').notNull().default(false),
    trackingId: text('tracking_id'),
    updatedBy: uuid('updated_by').references(() => users.id, { onDelete: 'set null' }),
    createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
    updatedAt: timestamp('updated_at', { withTimezone: true }).notNull().defaultNow(),
  },
);
```

### Columns

| Column | Type | Nullable | Default | Notes |
|---|---|---|---|---|
| `provider` | `analytics_provider` enum | NO | - | Primary key. One of the registered providers. |
| `enabled` | `boolean` | NO | `false` | Per-provider on/off switch. |
| `tracking_id` | `text` | YES | `NULL` | The vendor Tracking ID (e.g. Baidu's 32-char hex, GA's `G-XXXXXXXX`). Stored as plain text - not secret (it appears in public HTML by design). `NULL` means "not configured". |
| `updated_by` | `uuid` | YES | `NULL` | Foreign key to `users.id`. `ON DELETE SET NULL` so deleting an admin user does not cascade-delete the analytics config. |
| `created_at` | `timestamp with time zone` | NO | `now()` | Row creation time. |
| `updated_at` | `timestamp with time zone` | NO | `now()` | Last mutation time. |

### Indexes

- Primary key on `provider` (auto-created by Drizzle via `.primaryKey()`).
- No additional indexes needed - the table has at most a handful of rows
  (one per registered provider), and every access is by `provider` PK lookup.

### Relationships

- `updated_by` → `users.id` (many-to-one; `ON DELETE SET NULL`).

## Validation Rules

These are enforced at the service layer (`src/server/services/analytics.ts`)
and in the shared Zod schema (`packages/shared/src/analytics.ts`), not at the
DB constraint level (the DB only stores the raw `text`).

| Provider | Tracking ID pattern | Regex | Format hint (i18n) |
|---|---|---|---|
| `baidu_tongji` | 32-character hexadecimal string | `/^[a-f0-9]{32}$/i` | "32-character hex string" |
| `google_analytics` | GA4 Measurement ID | `/^G-[A-Z0-9]{6,12}$/` | "G-XXXXXXXX (e.g. G-A1B2C3D4E5)" |

**Service-layer validation**:
- A provider cannot be enabled with an empty or invalid Tracking ID. The
  service rejects the update with `DomainError('BAD_REQUEST', ...)` and
  preserves the previously active configuration.
- A Tracking ID can be saved for a disabled provider (so it can be re-enabled
  later without re-entering it). The validation only fires when `enabled =
  true`.
- Unknown provider keys (not in `analyticsProviderSchema`) are rejected by
  Zod at the API boundary before reaching the service.
- The Tracking ID regexes are strict enough to prevent any character that
  could break out of the script tag (no quotes, angle brackets, backslashes,
  or whitespace).

## State Transitions

Per-provider `enabled` state machine:

```
                 ┌──────────────────────────────────────┐
                 │                                      │
                 ▼                                      │
  ┌────────────────────┐  enable (valid Tracking ID)  ┌─────────┐
  │  disabled (no ID)  │ ──────────────────────────► │ enabled │
  └────────────────────┘                              └─────────┘
          │                                                │
          │ save Tracking ID (still disabled)              │ disable
          ▼                                                │
  ┌──────────────────────┐                                 │
  │ disabled (has ID)    │ ◄───────────────────────────────┘
  └──────────────────────┘
```

- `disabled (no ID)` → `disabled (has ID)`: save a Tracking ID while
  `enabled = false`. The ID is retained.
- `disabled (has ID)` → `enabled`: set `enabled = true`. Requires the saved
  Tracking ID to pass validation. If invalid, the transition is rejected.
- `enabled` → `disabled`: set `enabled = false`. The Tracking ID is retained
  for later re-enablement.
- Any `enabled` with empty/invalid Tracking ID is rejected; the row remains
  unchanged.

---

## Seed Data

No seed rows are inserted. The service treats a missing row as
`{ enabled: false, trackingId: null }` (the "disabled, no ID" state). This
matches the `content_data_source_settings` convention: rows are upserted on
first admin save.

## Migration

Generated by `pnpm db:generate` after editing `schema/enums.ts` and
`schema/index.ts`. The generated migration will:

1. `CREATE TYPE analytics_provider AS ENUM ('baidu_tongji', 'google_analytics');`
2. `CREATE TABLE analytics_provider_settings (...);`

The migration filename will be `0030_<random_name>.sql` (Drizzle's naming
convention). The corresponding `meta/0030_snapshot.json` is produced
automatically and MUST NOT be hand-edited.

After generation, run `pnpm db:generate` a second time to confirm "No schema
changes, nothing to migrate".

## Cache Interaction

The `analytics_provider_settings` table is read through an `unstable_cache`
wrapper tagged with `SITE_SHELL_CACHE_TAG` (300s revalidate), gated by
`shouldUseDataCache()` - exactly like `site_settings` and `system_themes`.

Every mutation (`updateAnalyticsProvider`, `upsertAnalyticsProviders`) calls
`invalidateSiteShellCache()` after the DB write, which `revalidateTag`s the
shell cache so the new provider state is visible on the next request.

## Impact on Existing Entities

None. The new table is additive; no existing table is modified. The
`users` table gains no new column (the `updated_by` reference is on the new
table only).

The `apiKeyScopeEnum` is NOT extended - analytics configuration is not
exposed to API keys (the `manage_appearance` action is already on the
API-key hard-deny list).

The `Action` and `Resource` unions in `permissions/index.ts` are NOT
extended - we reuse `manage_appearance` + `{ kind: 'appearance' }`.
