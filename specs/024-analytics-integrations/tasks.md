# Tasks: Web Analytics Integrations

**Input**: Design documents from `/specs/024-analytics-integrations/`

**Prerequisites**: plan.md (required), spec.md (required for user stories), research.md, data-model.md, contracts/

**Tests**: Tests are included as separate, explicitly marked tasks for every user story. The project's `CLAUDE.md` and `AGENTS.md` mandate "Always write unit tests and integration tests for new code changes" and "Verify code change by running tests and UI automations." Therefore tests are NOT optional for this feature.

**Organization**: Tasks are grouped by user story to enable independent implementation and testing of each story. Each user story maps to a phase; each phase is an independently shippable increment.

## Format: `[ID] [P?] [Story] Description`

- **[P]**: Can run in parallel (different files, no dependencies on incomplete tasks)
- **[Story]**: Which user story this task belongs to (e.g., US1, US2, US3, US4)
- Include exact file paths in descriptions

## Path Conventions

- **Monorepo** (pnpm + Turborepo):
  - App code: `apps/web/`
  - Shared schemas: `packages/shared/src/`
  - DB schema: `apps/web/src/server/db/schema/`
  - Migrations: `apps/web/src/server/db/migrations/` (generated by `pnpm db:generate`)
  - i18n catalogs: `apps/web/messages/{en,zh}.json`
- All paths below are project-relative.

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: No dependencies - can start immediately. T003→T004 must be sequential (enum before table that uses it); T005→T006 must be sequential (generate then verify).
- **Foundational (Phase 2)**: Depends on Phase 1 completion - BLOCKS all user stories. T007→T008→T009→T010→T011→T012→T013 are sequential (each builds on the previous in the same file). T014/T015 can run in parallel with each other but only after T013.
- **User Story 1 (Phase 3)**: Depends on Phase 2 completion. T019/T020 (i18n) and T016/T017 (tests) can run in parallel; T021 (API route) depends on the service from Phase 2; T022 (admin page) depends on T021; T023 (form) depends on T022; T024 (nav) depends on T019/T020.

- **User Story 3 (Phase 5)**: Depends on US2 (injection mechanism) and Phase 2 (cache invalidation). T029/T030 (tests) can run in parallel; T031/T032 are verification tasks that can run in parallel.
- **User Story 4 (Phase 6)**: Depends on Phase 2 (registry contract) and US2 (injection). T033/T034 (tests) can run in parallel; T035 (export) and T036 (docs) can run in parallel.
- **Polish (Phase 7)**: Depends on all user stories being complete. T037/T038/T039/T043 can run in parallel; T040/T041/T042 are sequential verification tasks.

### User Story Dependencies

- **User Story 1 (P1)**: Can start after Foundational (Phase 2). No dependencies on other stories. **This is the MVP** - an admin can configure providers even before US2 injects the scripts.
- **User Story 2 (P1)**: Depends on US1 (needs a configured provider to inject) and Phase 2 (service). Independently testable once US1 is done.
- **User Story 3 (P2)**: Depends on US2 (injection mechanism) and Phase 2 (cache invalidation). Primarily a verification phase - the cache logic is already built in Phase 2.
- **User Story 4 (P3)**: Depends on Phase 2 (registry) and US2 (injection). Primarily a verification + documentation phase - the pluggability is already designed into Phase 2.

- Services before API routes
- API routes before admin UI
- Admin UI before navigation entry
- Story complete before moving to next priority

## Implementation Strategy

### MVP First (User Story 1 Only)

1. Complete Phase 1: Setup (T001-T006) - shared schemas, enum, table, migration.
2. Complete Phase 2: Foundational (T007-T015) - service layer, cache, tests.

### Incremental Delivery

1. Setup + Foundational -> Foundation ready (service can read/write settings).
2. Add User Story 1 -> Test independently -> Deploy/Demo (MVP! Admin can configure providers.)
3. Add User Story 2 -> Test independently -> Deploy/Demo (Scripts appear on all pages.)

5. Add User Story 4 -> Test independently -> Deploy/Demo (Pluggability verified.)
6. Polish -> Final validation, OpenAPI, lint, full quickstart run.

### Parallel Team Strategy

With multiple developers:

1. Team completes Setup + Foundational together (T001-T015).
2. Once Foundational is done:
   - Developer A: User Story 1 (admin UI + API route)
   - Developer B: User Story 2 (layout injection + tests) - can start in parallel with US1 since it depends only on the service layer, but needs US1 to actually enable a provider for E2E
   - Developer C: User Story 3 + 4 (verification + documentation - lighter workload)
3. Stories complete and integrate independently; the only integration point is the root layout, which is owned by US2.
