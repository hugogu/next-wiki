# Implementation Plan: Web Analytics Integrations

**Branch**: `024-analytics-integrations` | **Date**: 2026-07-21 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/024-analytics-integrations/spec.md`

## Summary

Add a pluggable, framework-level web analytics integration layer. Administrators
configure analytics providers (Baidu Tongji, Google Analytics at launch) through
a dedicated admin page; each provider has its own Tracking ID and independent
enable/disable switch. The active providers' tracking scripts are injected into
the `<head>` of every rendered page through a single layout-level injection point
(the existing root `app/layout.tsx`), reusing the site-shell cache tag for ISR
revalidation - the same mechanism that already delivers the system theme CSS.
Public/anonymous pages keep their static/ISR representation; personalized
controls remain outside the cached document body. New providers can be added by
registering them through a bounded contract in the shared package + a service
registry, with zero page-component edits.

## Technical Context

**Language/Version**: TypeScript 5.6 on Node.js 20.9+ (Next.js 16 App Router runtime floor).

**Primary Dependencies**: Next.js 16 (App Router, `unstable_cache`, `revalidateTag`,
`revalidatePath`), React 19.2, Drizzle ORM (PostgreSQL schema + migrations),
Zod (shared schemas), `@next-wiki/shared` (cross-package contracts), Tailwind CSS
+ CSS-variable tokens (admin UI), existing in-house UI primitives in
`src/components/ui/`.

**Storage**: PostgreSQL 16. One new table `analytics_provider_settings` keyed by
`provider` (pgEnum `analytics_provider`, values `baidu_tongji`,
`google_analytics`), with `enabled`, `tracking_id`, and audit columns. Mirrors the
`content_data_source_settings` key-addressed registry pattern.

**Testing**: Vitest (unit + integration) for the service, validation, script
template builder, and cache invalidation. Playwright E2E for the admin page and
script-injection assertion on multiple surfaces (reader, editor, admin, public
home). Reuse the existing `shouldUseDataCache()` test-bypass guard so tests run
without the data cache.

**Target Platform**: Linux server (Docker Compose / Kubernetes), browser clients.
Tracking scripts are vendor-provided browser snippets (Baidu Tongji `hm.js`,
Google Analytics `gtag/js`), rendered server-side as raw HTML strings and inlined
via `dangerouslySetInnerHTML` (the same mechanism the root layout already uses
for `themeScript`).

**Project Type**: Web service / monorepo (`apps/web` Next.js app inside the pnpm
+ Turborepo workspace). This feature touches only `apps/web` and
`packages/shared`.

**Performance Goals**: The cached site-shell lookup for analytics scripts must
add no perceptible latency to first byte on cache hit (data served from the
existing `SITE_SHELL_CACHE_TAG`-tagged `unstable_cache`, 300s revalidate, same as
`getActiveThemeCss`). On cache miss the cost is one indexed DB row lookup
(`provider` is the primary key). Tracking scripts are delivered in the initial
HTML; no client-side round-trip is added beyond what the vendor script itself
performs.

**Constraints**:
- Tracking IDs are **not secret** (they appear in public HTML by design) - they
  MUST NOT be encrypted; the `encryptKey`/`decryptKey` modules stay unused for
  this feature.
- Public/anonymous pages MUST remain static/ISR-compatible (constitutional
  mandate P12). The analytics script is the same for every visitor and lives in
  the cached document body; no session/cookie/header read may vary the script.
- Admin mutations MUST invalidate the site-shell cache so the new script set is
  visible on the next request, with no 5-minute staleness window in production.
- The injection point MUST be the single root `app/layout.tsx` `<head>` block; no
  page component may inject analytics directly.
- New providers MUST be registered through the bounded contract (shared-package
  enum + service registry entry). Unknown provider keys are rejected at the
  service layer (constitutional P10 - explicit over implicit).
- Migrations MUST be generated by `pnpm db:generate`; never hand-author the
  `.sql`, `meta/_journal.json`, or `meta/NNNN_snapshot.json`.

**Scale/Scope**: Two built-in providers at launch (Baidu Tongji, Google
Analytics). The contract is designed so additional providers (Matomo, Plausible,
Umami, etc.) can be added by appending to the enum + registry with no page-code
edits. Per-page impact: zero. Admin UI: one new page with one form. Public API
surface: one new public-readable `GET /api/settings/analytics` endpoint and one
admin-only `PUT /api/settings/analytics` endpoint.

### Public Content Delivery

The feature **does** change anonymously readable published content: it adds one
or more third-party analytics `<script>` tags to the `<head>` of public pages.

- **Static/ISR representation**: The analytics script string is read from the
  site-shell cache (`unstable_cache` tagged `SITE_SHELL_CACHE_TAG`, 300s
  revalidate - the same cache that already serves `getActiveThemeCss` and
  `getSiteView`). The root `app/layout.tsx` inlines the script string via
  `dangerouslySetInnerHTML` directly into the cached document body. No DB query,
  session lookup, cookie read, or request-header read happens per visitor.
- **Cache lifetime**: 300s revalidate (matches the existing shell-cache
  convention), plus on-demand invalidation on mutation.
- **Exact invalidation paths/tags**: `invalidateSiteShellCache()` is called on
  every `updateAnalyticsProvider*` mutation. This `revalidateTag(SITE_SHELL_CACHE_TAG, 'max')`
  revalidates the layout representation, which propagates to all
  `revalidate: 'layout'`-cached pages (the same mechanism used by site identity
  and system theme changes).
- **Personalized controls**: The admin configuration form lives at
  `/admin/analytics` (admin-only, fully dynamic). No admin-only or personalized
  content is ever injected into the public document body; the script set is
  identical for every visitor.

**Structure Decision**: This is a Next.js App Router monorepo; we add a new admin
page + API route under the existing `(admin)` route group and `api/settings/`
collection, plus one new service file and one new shared schema file. The only
layout-level change is the root `app/layout.tsx` `<head>` injection. The schema
change is additive (one new table + one new enum). All directories follow the
existing convention (`src/server/services/`, `src/server/db/schema/`,
`src/components/admin/`, `packages/shared/src/`).
